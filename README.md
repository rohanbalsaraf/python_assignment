# FastBox Logistics Simulator

## Engineering Decisions & Assumptions

This document outlines the core engineering decisions and mathematical logic assumptions made while developing the `simulator.py` logistics script.

### 1. Dynamic Route Tracking (The Continuous Path)
When calculating the `total_distance` traveled by an agent during a "day of operations," it was unrealistic to assume the agent "teleported" back to their home base before picking up every single package. 
* **Decision:** I implemented dynamic state-tracking. An agent begins at their home base, travels to a warehouse, and drops the package at the destination. Crucially, their **current location** is then updated to that destination, making it their new starting point for traveling to the next warehouse. This perfectly simulates a realistic, continuous daily delivery route rather than isolated, independent trips.

### 2. Tie-Breaking & Sequential Processing (FIFO)
When assigning packages to the nearest agent, multiple agents could technically be equidistant from a warehouse. 
* **Decision:** The algorithm uses a strict "less than" (`<`) mathematical operator when comparing Euclidean distances. This assumes a First-In, First-Out (FIFO) tie-breaker—if there is a tie, the package is assigned to whichever agent was processed first in the JSON data. The deliveries themselves are also processed in the exact sequential order they appear in the data without attempting to run a complex Traveling Salesperson Problem (TSP) optimization.

### 3. Edge-Case Handling (The "Infinity" JSON Bug)
Agents located too far from warehouses are often assigned zero packages. When calculating their efficiency (`total_distance / packages_delivered`), this causes a division-by-zero crash.
* **Decision:** To keep the internal math robust, idle agents are assigned an efficiency of Python's `float('inf')` so they are mathematically excluded from being selected as the `best_agent`. However, because strict JSON specifications (RFC 8259) do not accept `Infinity` as a valid number, the reporting logic intercepts this value and outputs a clean `0` to the final `report.json` to ensure 100% strict JSON serialization compliance.

### 4. Safe Batch Processing Architecture
The script is capable of processing an entire folder of test cases via batch mode.
* **Decision:** To prevent the batch loop from constantly overwriting the core `report.json` and `performer.csv` files generated by a single base-case run, the script utilizes an `is_batch` flag. When running a folder, the script safely isolates the outputs to `batch_report.json` and `batch_summary.csv` to protect the integrity of the single-file reports required for submission.

### 5. Bonus Implementations
* **Time vs. Distance (Delays):** When running with the `--delays` flag, it is assumed that 1 unit of Euclidean distance mathematically equates to 1 unit of base time. Random delays are generated and added on top of this base time, and tracked entirely separately under a new `total_time` variable in the output JSON.
* **ASCII Visualization:** Running with the `--ascii` flag triggers a text-based visual map that dynamically prints the sequential steps (with distances and delays) of every agent's continuous daily route.
* **Mid-day Agent:** Running with `--midday-agent` splits the deliveries in half and dynamically injects a new agent (`A_NEW`) at coordinates `[50, 50]` to handle the afternoon queue.
